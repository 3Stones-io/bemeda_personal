üêô GitHub-First Architecture Overview

USE is built on the principle that GitHub Issues are your database. Every scenario, every actor, every piece of requirements lives in GitHub Issues with structured metadata, making them searchable, trackable, and automatically processable.

Complete GitHub Integration Flow

1
Create USE Scenario

Business stakeholder creates GitHub Issue using enforced template with all required fields

2
Auto-Validation

GitHub Actions validate Gherkin syntax, check actor consistency, and add appropriate labels

3
Generate Registry

Script generates MASTER-REGISTRY.json from all component issues for system-wide tracking

4
Update Documentation

Automated scripts regenerate documentation pages, sitemaps, and navigation from registry

5
Deploy to Pages

GitHub Pages automatically builds and deploys updated documentation with live scenario data

6
Team Collaboration

Tech and UX teams comment, reference parallel scenarios, and track implementation progress

How Templates Enforce Quality

Required Fields

Templates make essential fields mandatory, preventing incomplete scenarios.

  • Component Type must be selected
  • Gherkin scenario is required
  • Actor information is mandatory

Role-Specific Workflows

üíº

Product Owner / Business Stakeholder

Focuses on business scenarios (B_S) that deliver value to users and the organization.

  • Create B_S scenarios using GitHub template
  • Define success criteria and business metrics
  • Reference parallel T_S and U_S scenarios
  • Review and approve implementation progress
  • Use GitHub Project boards for prioritization
  • Comment on technical feasibility discussions
‚öôÔ∏è

Tech Lead / Developer

Creates technical scenarios (T_S) that implement the infrastructure needed for business scenarios.

  • Create T_S scenarios for technical implementation
  • Reference parent B_S scenarios for context
  • Define API contracts and data structures
  • Set up automated BDD tests using step definitions
  • Update scenario status as implementation progresses
  • Provide technical feasibility feedback on B_S scenarios
üé®

UX Designer

Creates UX scenarios (U_S) that define how users interact with the system through interfaces.

  • Create U_S scenarios for user interface flows
  • Define accessibility and responsive design requirements
  • Reference both B_S and T_S scenarios for context
  • Specify interaction patterns and visual feedback
  • Create wireframes and prototypes linked to scenarios
  • Validate UX scenarios with user testing

üîß Troubleshooting and Maintenance

Common issues you'll encounter and how to resolve them quickly.

Common Issues and Solutions

Issue: Automation Workflow Not Triggering

# Check workflow status gh workflow list gh workflow view "BDD Scenario Automation" # Check recent runs gh run list --workflow="bdd-scenario-automation.yml" --limit 5 # View detailed logs gh run view [RUN_ID] --log Common causes: ‚Ä¢ Workflow permissions not set correctly ‚Ä¢ Issue doesn't have required labels ‚Ä¢ Template format changed without updating workflow

Issue: Component Registry Not Updating

# Manually trigger registry generation node scripts/generate-component-map.js # Check if GitHub CLI is authenticated gh auth status # Verify component issues have correct labels gh issue list --label "component" --state all Troubleshooting steps: ‚Ä¢ Ensure GitHub CLI token has repo and project permissions ‚Ä¢ Check that issues have "component" label ‚Ä¢ Verify MASTER-REGISTRY.json is writable

Issue: Parallel Scenarios Not Linking Correctly

‚ö†Ô∏è Parallel Scenario Reference Format

Make sure parallel scenarios are referenced exactly as: | T_S001_US001 | Technical | Expected result | in a table format within the Gherkin scenario. The automation looks for this specific pattern.
